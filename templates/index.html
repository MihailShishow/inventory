Описание API


Как работает API?

Учет движения товаров и кассовых операций производится с использованием документов: накладных и кассовых ордеров. Каждая операция приема/выдачи должна быть зарегистрирована для облегчения учета ресурсов и сведения бухгалтерского баланса.

Зарегистрированный документ нельзя удалить и изменить.

Учетные данные вычисляются автоматически и доступны после регистрации соответствующих документов.

Учетные данные различных ресурсов:

  • Product: in_stock - текущий остаток товаров на складе;
  • Counterparty: saldo - разница между приходом и расходом данного контрагента;
  • Income/OutcomeInvoice: total - сумма накладной.

Описание ресурсов

Product - единица товара. Содержит информацию: наименование, единица измерения, цена, текущий остаток. Для удобства может быть отнесен к какой-нибудь категории.

Counterparty - контрагент, в данном случае участник кассовых и/или товарообменных операций. Содержит информацию: название, сальдо.

ProductCategory - категория продуктов. Не является обязательным параметром. Содержит информацию: название.

IncomeWarrant - приходный кассовый ордер. Содержит информацию: номер, дату, сумму, контрагента, комментарий, регистрацию.

OutcomeWarrant - расходный кассовый ордер. Содержит информацию: номер, дату, сумму, контрагента, комментарий, регистрацию.

IncomeInvoice - приходная накладная. Содержит информацию: номер, дату, сумму, контрагента, регистрацию, список товаров.

OutcomeInvoice - расходная накладная. Содержит информацию: номер, дату, сумму, контрагента, регистрацию, список товаров.

ProductIncome - товарный приход. Содержит информацию: товар, цена, количество, сумма, накладная.

ProductOutcome - товарный расход. Содержит информацию: товар, цена, количество, сумма, накладная.

Предположим, на склад прибыл кошачий корм в количестве 100 шт. от поставщика "Охель Хатулим" (название выдумано, переводится с иврита как кошачий корм). Кладовщик проверил фактический приход, он совпал с расходной накладной поставщика. Теперь нужно оприходовать поступивший товар. Для этого нужно создать приходную накладную. Кладовщик вводит номер накладной, дату и вдруг обнаруживает, что данного поставшика нет в списке контрагентов. Следовательно, его нужно добавить. Кладовщик создает нового контрагента, вводя название предприятия поставщика. Всё, можно дальше создавать накладную. Наш усердный работник указывает, от кого прибыл товар. Далее, он создает новую запись в приходной накладной, но данного товара также нет в списке. Поэтому, его также необходимо создать. Кладовщик вводит имя товара, единицу измерения, высчитывает наценку и устанавливает новую цену для продажи. Теперь можно продолжить ввод данных в накладную. Накладная создана, данные введены, но результат не отображается! Чтобы приход товара был учтен окончательно, необходимо зарегистрировать документ.


Последовательность действий кладовщика:


Создание (POST) контрагента (Counterparty) -> создание (POST) приходной накладной (IncomeInvoice) -> создание (POST) товара (Product) -> создание (POST) товарного прихода (ProductIncome) -> регистрация (PATCH) приходной накладной (IncomeInvoice).


После данных операций при получении (GET) свежесозданных ресурсов кладовщик видит, что сальдо (saldo) контрагента (Counterparty), соответствующего поставщику кошачьего корма изменилось на сумму товарного прихода по закупочной цене со знаком минус, итоговая сумма (total) приходной накладной (IncomeInvoice) также была высчитана автоматически, а ресурс товара (Product) показывает количество товара в наличии (in_stock).
